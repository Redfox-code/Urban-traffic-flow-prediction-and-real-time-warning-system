# 架构决策记录 (ADR)

> 格式：日期、状态、背景、决策、后果。只记录不可逆的关键决策。

---

## ADR-001：选择Flask而非Django
- **日期**：2026-06-29
- **状态**：✅ 已确认
- **背景**：需要轻量级后端，项目工期14天紧，Django学习成本高、约定太多
- **决策**：使用 Flask 3.x + 手动组装SQLAlchemy/JWT/Celery
- **后果**：
  - ✅ 灵活性高，每个组件可独立选型
  - ✅ 项目结构简洁，上手快
  - ⚠️ 需自行保证各组件版本兼容性

## ADR-002：KNN + 随机森林双模型（不做深度学习）
- **日期**：2026-06-29
- **状态**：✅ 已确认
- **背景**：短时交通流量预测。深度学习（LSTM/GRU）效果好但需要大量数据和调参时间
- **决策**：KNN回归作为简单可靠baseline + 随机森林处理非线性特征
- **后果**：
  - ✅ 训练快、可解释、调参简单
  - ✅ 14天工期可完成
  - ⚠️ 极限精度不如深度学习，但在课程项目场景下足够

## ADR-003：前后端完全分离
- **日期**：2026-06-29
- **状态**：✅ 已确认
- **背景**：前端Vue 3，后端Flask，需要确定通信方式
- **决策**：完全分离，RESTful API + WebSocket，Vite开发代理到Flask
- **后果**：
  - ✅ 前后端可并行开发
  - ✅ 部署灵活
  - ⚠️ 需处理CORS
  - ⚠️ 需维护API文档

## ADR-004：WebSocket用于预警推送而非轮询
- **日期**：2026-06-29
- **状态**：✅ 已确认
- **背景**：预警需要实时性（秒级），传统HTTP轮询延迟高、浪费带宽
- **决策**：Flask-SocketIO实现WebSocket，前端Socket.IO Client接收
- **后果**：
  - ✅ 延迟 < 1秒
  - ✅ 节省带宽
  - ⚠️ 需处理断线重连、心跳保活
  - ⚠️ 部署时需确认服务器支持WebSocket

## ADR-005：五Agent循环工程协作模式
- **日期**：2026-07-01
- **状态**：✅ 已确认
- **背景**：5位小组成员各有分工，需要一套高效的AI协作机制
- **决策**：采用 Addy Osmani Loop Engineering 六大构建块，5个Implementer Agent + 1个Judge Agent，文件通信、看板协调
- **后果**：
  - ✅ 角色清晰，职责不冲突
  - ✅ 独立审查保证质量
  - ⚠️ 文件通信有延迟（下一轮心跳才能发现），但对课程项目影响不大
