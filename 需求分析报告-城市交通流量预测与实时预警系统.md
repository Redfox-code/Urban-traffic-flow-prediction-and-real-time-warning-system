## 摘要

随着我国城市机动车保有量的持续增长，交通拥堵已成为制约城市运行效率的突出问题。传统交通管理方式依赖人工经验和固定配时信号灯，缺乏对交通流量的**实时感知**与**前瞻性预测**能力，往往在拥堵发生后才被动响应，错失最佳疏导时机。

本报告围绕"**城市交通流量预测与实时预警系统**"展开需求分析。系统基于 **Python Flask** 轻量级 Web 框架、**SUMO（Simulation of Urban MObility）微观交通仿真平台**、**Scikit-learn 机器学习库**（KNN 回归 + 随机森林）以及 **Vue 3 + ECharts + 高德地图** 前端技术栈，构建一套集**SUMO微观交通仿真、短时流量预测、拥堵实时预警、最优路径规划**于一体的智能交通管理平台。系统面向两类用户：交通管理者可通过实时路况监控大屏和预警看板快速掌握全城交通态势并下发调度指令；数据分析员可通过预测模型管理和历史数据回溯持续优化预测精度。

本报告按照课程要求，完成了以下分析工作：

1. **需求分析**：梳理了系统6大功能模块（数据采集、流量预测、实时监控、拥堵预警、路径规划、系统管理）的功能性需求，以及性能、安全性、可用性等非功能性需求。
2. **可行性分析**：从经济、技术、社会三个维度论证了项目的可行性——技术栈开源免费、开发周期可控（14天）、符合国家智慧城市建设政策导向。
3. **结构化分析（SA）**：绘制了数据流图（DFD，包含顶层图/0层图/1层图）、实体关系图（E-R图，8个核心实体）、状态转换图（STD，路段交通状态机），并编写了详细的数据字典。
4. **面向对象分析（OOA）**：使用 UML 2.0 绘制了用例图（3角色30+用例）、类图（MVC三层架构）、顺序图（预测流程/预警流程）、活动图（数据Pipeline/预警处理泳道图）。

本报告为后续的**概要设计**与**详细设计**阶段提供了完整的需求基线。小组共5人，按后端架构、算法工程、前端开发、地图可视化、测试文档五个方向分工协作，计划在14天内完成从需求分析到系统交付的全流程。

## 二、项目背景

### 2.1 城市交通拥堵问题日益严峻

随着我国城镇化进程的加速和机动车保有量的持续增长，城市交通拥堵已成为影响居民出行效率和生活质量的突出问题。据公安部统计，截至2025年底，全国机动车保有量已超过4.5亿辆，其中汽车保有量超过3.5亿辆。北京、上海、广州、深圳等超大城市的日均拥堵时长超过2小时，交通拥堵造成的经济损失约占GDP的1%-3%。

传统的交通管理方式主要依赖人工经验和固定配时信号灯，缺乏对交通流量变化的**实时感知**和**前瞻性预测**能力。当拥堵已经发生时再采取措施，往往错过了最佳的疏导时机。

### 2.2 智能交通系统（ITS）的发展需求

智能运输系统（Intelligent Transportation System，ITS）是将先进的信息技术、数据通信技术、传感器技术、电子控制技术以及计算机处理技术等综合运用于整个交通运输管理体系，建立起的一种在大范围内、全方位发挥作用的实时、准确、高效的综合运输和管理系统。

交通流量预测是ITS的核心技术之一。准确的短时交通流量预测能够：

- **支撑交通信号灯动态优化**：根据预测流量调整绿信比
- **辅助出行路径规划**：为用户推荐避开拥堵的最优路线
- **实现拥堵提前预警**：在拥堵形成前15-30分钟发出预警
- **服务城市交通规划**：基于流量趋势数据支撑路网扩容决策

### 2.3 项目目标

本项目旨在构建一个**城市交通流量预测与实时预警系统**，以Web应用的形式，实现以下核心目标：

1. **SUMO微观交通仿真**：基于 SUMO（Simulation of Urban MObility）构建城市路网模型，定义车流需求，部署虚拟检测器，通过微观仿真生成高保真的多路段、多时段交通流量数据，替代简单的公式模拟器
2. **短时流量预测**：基于机器学习算法（KNN/随机森林），对未来15/30/60分钟的路段交通流量进行预测
3. **拥堵实时预警**：当预测流量超过预设阈值时，通过WebSocket向前端推送实时预警
4. **路况可视化展示**：基于高德地图展示实时路况热力图，以及历史与预测流量的对比图表；SUMO-GUI 提供2D交通仿真动画作为辅助展示
5. **最优路径规划**：基于实时和预测路况，使用Dijkstra算法计算最短通行时间路径

## 三、研究现状

### 3.1 交通流量预测方法综述

交通流量预测研究经历了从经典统计方法到机器学习、再到深度学习的演进历程。

**（1）经典统计方法**

早期研究主要采用时间序列分析模型，如ARIMA（自回归积分滑动平均模型）及其变体（SARIMA、ARIMAX）。这类方法假设交通流量具有平稳性和线性关系，在短时预测中表现尚可，但难以捕捉交通流中的非线性突变特征。此外，卡尔曼滤波（Kalman Filter）也被广泛应用于交通状态估计，但对历史数据质量和模型参数标定要求较高。

**（2）机器学习方法**

随着数据采集能力的提升，基于机器学习的方法逐渐成为主流。K近邻（K-Nearest Neighbors，KNN）是一种经典的非参数回归方法，通过查找历史中与当前交通状态最相似的K个时刻，加权平均得到预测值。KNN实现简单、可解释性强，无需显式训练阶段，适合工程落地。支持向量回归（SVR）、随机森林（Random Forest）、梯度提升树（XGBoost）等集成学习方法也被广泛采用，在交通流量预测中取得了较好的精度。

**（3）深度学习方法**

近年来，LSTM（长短期记忆网络）、GRU（门控循环单元）、Transformer等深度学习模型被引入交通预测领域。这些模型能够更好地捕捉时间序列中的长期依赖关系和非线性特征。图神经网络（GNN）结合路网拓扑结构进行时空联合预测也是一个前沿方向。但深度学习方法通常需要大量训练数据和计算资源，对课程项目而言门槛较高。

### 3.2 现有系统调研

**（1）开源项目 Traffic Monitor**

GitHub 上的开源项目 [Mike014/Traffic-Monitor](https://github.com/Mike014/Traffic-Monitor) 是一个基于 Flask + KNN + Dijkstra 的交通监测系统。用户在地图上点击选择起点和终点，系统计算最短路径并预测沿线的交通状况。该项目的核心架构（Flask后端 + KNN预测 + Dijkstra路径规划）为本系统提供了直接的工程参考。

**（2）交通预测项目**

[Sehan-D-Amendra/Traffic-prediction-project](https://github.com/Sehan-D-Amendra/Traffic-prediction-project) 采用 Flask + Scikit-learn 构建，用户通过Web表单输入日期、时间、天气等参数，系统输出预测的交通流量和类型。该项目的ML pipeline设计（数据预处理→特征工程→模型训练→Web部署）值得借鉴。

**（3）商业系统对比**

百度地图、高德地图等商业导航应用已具备路况展示和未来出行时间预估功能，但其技术细节不公开。本项目的定位是教学和技术验证性质，重在展示ITS核心技术原理。

### 3.3 本项目的创新与定位

相比现有开源项目，本系统的特色在于：

- **SUMO微观仿真替代简单模拟**：使用工业级交通仿真工具 SUMO 替代自编公式生成数据。SUMO 内置跟驰模型（Car-Following Model）和换道模型（Lane-Changing Model），生成的交通数据更接近真实场景，且学术界广泛认可。路网可从 OpenStreetMap 直接导入真实城市路网（如北京交通大学周边区域），车流定义支持分时段密度调整
- **多模型集成**：同时实现KNN回归和随机森林两种预测算法，支持模型性能对比
- **实时预警推送**：引入WebSocket实现预测结果的实时推送，而非被动查询
- **完整的数据管道**：从SUMO仿真→XML数据解析→MySQL存储→模型训练→预测API→可视化展示的完整闭环
- **SUMO-GUI 实时仿真动画**：可同步展示 SUMO-GUI 的2D微观交通仿真画面，车辆移动、信号灯变化一目了然，演示效果远超静态图表

## 四、需求分析

### 4.1 功能性需求

本系统的功能性需求按用户角色和功能模块两个维度进行划分。

#### 4.1.1 用户角色定义

| 角色 | 描述 | 核心职责 |
|------|------|----------|
| **交通管理员** | 城市交通管理中心的操作人员 | 查看实时路况、接收和处理预警、配置系统参数、查看统计报表 |
| **数据分析员** | 交通数据分析师 | 训练和评估预测模型、查看预测性能指标、导出分析数据 |
| **普通出行者** | 使用系统的公众用户 | 查询路况、规划出行路径、查看交通预测 |

#### 4.1.2 功能模块划分

**模块一：交通数据管理模块**

| 编号 | 功能名称 | 功能描述 | 优先级 |
|------|----------|----------|--------|
| F-DM-01 | SUMO仿真数据生成 | 使用 SUMO 搭建城市路网（从 OpenStreetMap 导入或手绘 .net.xml），定义分时段车流需求（.rou.xml），部署 E2 感应线圈检测器（.det.xml），运行 SUMO 命令行仿真或通过 TraCI Python API 实时控制仿真进程，输出各检测器每分钟的流量、速度、占有率数据 | 高 |
| F-DM-02 | 数据导入导出 | 支持CSV/JSON格式的流量数据导入和导出，可从SUMO输出XML中解析数据写入MySQL，也可将外部真实数据集导入 | 中 |
| F-DM-03 | 数据预处理 | 对SUMO仿真输出的原始流量数据进行清洗（缺失值填充、异常值检测与修正）、标准化（Min-Max归一化或Z-score标准化）、特征工程（时间特征提取、滞后特征构造） | 高 |
| F-DM-04 | 路段信息管理 | 对路网中的路段进行CRUD（增删改查）管理，包括路段名称、起点经纬度、终点经纬度、长度、车道数、限速、所属区域等属性 | 高 |
| F-DM-05 | 检测器信息管理 | 对虚拟交通检测器（数据采集点）进行管理，关联到具体路段，配置检测器类型和状态 | 中 |

**模块二：流量预测模块（核心）**

| 编号 | 功能名称 | 功能描述 | 优先级 |
|------|----------|----------|--------|
| F-PR-01 | KNN流量预测 | 使用K近邻算法，基于历史流量数据对指定路段未来15/30/60分钟的交通流量进行预测。支持调整K值、距离度量方式（欧氏距离/曼哈顿距离）、加权方式等参数 | 高 |
| F-PR-02 | 随机森林预测 | 使用随机森林回归算法进行交通流量预测，支持调整树的数量（n_estimators）、最大深度等超参数 | 高 |
| F-PR-03 | 拥堵等级分类 | 基于预测的流量和速度数据，将路段交通状态分为四级：畅通（绿色）、缓行（黄色）、拥堵（橙色）、严重拥堵（红色），各等级阈值可配置 | 高 |
| F-PR-04 | 模型评估对比 | 在同一测试集上计算并展示两种模型的MAE（平均绝对误差）、RMSE（均方根误差）、MAPE（平均绝对百分比误差）等指标，支持可视化对比（预测值vs真实值曲线图） | 高 |
| F-PR-05 | 模型定时重训练 | 通过Celery定时任务，每天凌晨自动使用最新的历史数据重新训练模型，保证预测精度不退化 | 中 |
| F-PR-06 | 模型持久化 | 使用joblib/pickle将训练好的模型序列化保存，预测时直接加载，避免每次预测都重新训练 | 高 |

**模块三：实时监控与预警模块**

| 编号 | 功能名称 | 功能描述 | 优先级 |
|------|----------|----------|--------|
| F-RM-01 | 实时路况地图 | 基于高德地图JavaScript API，在地图上以不同颜色展示各路段的实时交通状态（畅通/缓行/拥堵/严重拥堵），颜色随数据自动更新 | 高 |

**（2）核心算法可行性**

- **KNN预测**：非参数方法，无需显式训练，只需存储历史数据即可预测。在Python中用Scikit-learn的`KNeighborsRegressor`仅需数行代码即可实现。参考项目Traffic-Monitor已验证此技术路线可行。
- **随机森林预测**：Scikit-learn的`RandomForestRegressor`是成熟的即用型实现，调参直观，对交通数据的非线性特征捕捉能力强。
- **Dijkstra路径规划**：经典图算法，Python实现简洁。高德地图API自身也提供了路径规划接口可作为备选方案。
- **SUMO交通仿真**：SUMO 是德国航空航天中心（DLR）开发的开源微观交通仿真平台，在ITS学术界和工业界被广泛使用。其内置的跟驰模型（Krauss/IDM）和换道模型能够高保真地模拟真实车辆行为。Python 通过 TraCI 接口可直接控制仿真进程，通过 sumolib 库可解析输出文件。从 OpenStreetMap 导入真实路网只需一条命令，部署 E2 检测器后即可输出标准格式的流量/速度/占有率数据。SUMO 完全开源免费（EPL-2.0 许可证），Windows/macOS/Linux 全平台支持。

#### 6.4.4 处理逻辑

| 编号 | 处理名 | 输入数据流 | 输出数据流 | 处理逻辑简述 |
|------|--------|------------|------------|-------------|
| PL-01 | P3 KNN流量预测 | 历史流量特征 + 当前状态特征 | 预测流量值 | 1.加载历史流量特征矩阵；2.计算当前特征与历史特征的欧氏距离；3.选取K个最近邻样本；4.加权平均得到预测流量值 |
| PL-02 | P3 拥堵等级分类 | 预测流量 + 预测速度 | 拥堵等级 | 1.读取当前配置的阈值；2.比较预测值与阈值；3.输出对应的拥堵等级枚举值 |
| PL-03 | P4 拥堵预警判断 | 预测结果(拥堵等级≥拥堵) | 预警事件 | 1.检查预测拥堵等级是否达到"拥堵"或以上；2.检查该路段最近N分钟内是否已有相同预警（去重）；3.满足条件则创建预警事件记录；4.通过WebSocket推送预警 |
| PL-04 | P6 Dijkstra路径规划 | 起点 + 终点 + 路网图(路段为边，通行时间为权重) | 最短时间路径 | 1.将路网建模为加权有向图G(V,E)，权重=路段长度/预测速度；2.运行Dijkstra算法；3.返回最短时间路径的节点序列 |
| PL-05 | P2.3 数据预处理 | 原始流量记录 | 清洗后数据 | 1.缺失值检测→使用前后均值填充；2.3σ异常值检测→替换为均值；3.Min-Max归一化到[0,1]区间；4.构造滞后特征(t-1, t-2, t-3时刻的流量作为额外特征) |

### 9.2 开发方法论

#### 9.2.1 采用"敏捷开发 + 增量迭代"模式

本项目虽然规模不大，但时间紧迫（14天），适合采用**简化版敏捷开发**方法：

| 敏捷实践 | 本项目中的具体应用 |
|----------|-------------------|
| **Sprint（迭代周期）** | 每个阶段为一个 Sprint（需求→概要→开发→交付），共4个 Sprint |
| **Daily Standup（每日站会）** | 每天10分钟，线上或线下：昨天做了什么？今天做什么？遇到什么阻碍？|
| **Sprint Review（评审）** | 每个阶段结束时进行 PPT 展示，即阶段性评审 |
| **Backlog（任务清单）** | 使用 GitHub Issues / 腾讯文档维护功能需求清单，标注优先级 |
| **Pair Programming（结对编程）** | 算法与后端在模型集成时结对、前端主副在复杂页面（地图）结对 |
| **CI（持续集成）** | 不强制搭建 CI 服务器，但要求每人每天至少 push 一次代码到 GitHub |

#### 9.2.3 API 接口设计规范

**统一响应格式**：

```json
{
  "code": 200,           // HTTP状态码
  "message": "success",  // 提示信息
  "data": {},            // 响应数据（可以是对象或数组）
  "timestamp": 1719676800
}
```

**RESTful 命名规范**：

| 方法 | URL 模式 | 示例 | 说明 |
|------|----------|------|------|
| GET | `/api/{resource}` | `/api/sections` | 获取列表 |
| GET | `/api/{resource}/{id}` | `/api/sections/5` | 获取详情 |
| POST | `/api/{resource}` | `/api/sections` | 创建资源 |
| PUT | `/api/{resource}/{id}` | `/api/sections/5` | 更新资源 |
| DELETE | `/api/{resource}/{id}` | `/api/sections/5` | 删除资源 |
| POST | `/api/{resource}/{id}/{action}` | `/api/predict/5/forecast` | 特定操作 |

**核心 API 列表（概要预览）**：

| 模块 | 方法 | URL | 说明 |
|------|------|-----|------|
| 认证 | POST | `/api/auth/login` | 登录 |
| 认证 | POST | `/api/auth/register` | 注册 |
| 路段 | GET/POST | `/api/sections` | 路段列表/新增 |
| 路段 | PUT/DELETE | `/api/sections/{id}` | 路段更新/删除 |
| 流量数据 | GET | `/api/traffic/records` | 流量记录查询(支持时间/路段筛选) |
| 预测 | POST | `/api/predict/forecast` | 执行流量预测 |
| 预测 | GET | `/api/predict/results` | 查询历史预测结果 |
| 预测 | POST | `/api/predict/train` | 触发模型训练 |
| 预警 | GET | `/api/warnings` | 预警列表 |
| 预警 | PUT | `/api/warnings/{id}/process` | 处理预警 |
| 路径规划 | POST | `/api/route/plan` | 计算最优路径 |
| 统计 | GET | `/api/stats/daily` | 日报统计 |
| 统计 | GET | `/api/stats/ranking` | 拥堵排行 |

### 9.5 风险与应对

| 风险项 | 概率 | 影响 | 应对预案 |
|--------|------|------|----------|
| **SUMO 环境配置失败** | 低 | 高 | Windows 下可使用 conda 一键安装；备选方案：使用 SUMO 官方 Docker 镜像；极端情况：退回自写 Python 模拟器（仍可完成系统） |
| **预测模型精度不达标** | 中 | 中 | SUMO 仿真数据规律性可控（高峰/平峰/夜间差异明显），模型容易学到合理模式；降级方案：使用历史均值法/滑动平均法替代ML |
| **高德地图API集成困难** | 低 | 中 | 高德官方文档+示例代码齐全；备选方案：使用开源的 Leaflet.js + OpenStreetMap，无需API Key |
| **前后端联调时间不足** | 中 | 高 | 前端先用Mock数据独立开发；D10全天为联调缓冲；核心功能(预测+预警)优先联调，统计页面可降级 |
| **Redis 在 Windows 下安装困难** | 中 | 低 | 使用 Memurai(Windows版Redis) 或 Docker Desktop 运行Redis；极端情况：先用内存队列替代Celery+Redis |
| **某成员进度严重落后** | 低 | 高 | 每日站会及时发现；相邻角色成员支援（后端帮算法、前端主帮辅）；文档工程师可协助代码测试 |
| **演示视频录制超时** | 低 | 中 | 提前写好脚本，控制在5分钟以内；展示核心流程（登录→地图→预测→预警→路径规划→报表）+ SUMO-GUI 仿真动画，管理功能快进 |
| **GitHub 代码冲突** | 低 | 低 | 每人独立分支开发、合并前Pull Request Review；前后端分开目录，冲突概率低 |
