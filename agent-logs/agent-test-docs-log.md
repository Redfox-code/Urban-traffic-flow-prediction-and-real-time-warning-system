## 思考轨迹

### D3-T05：数据库设计与E-R图（2026-07-01）

**我理解的任务**：在Leader提供的8个实体清单基础上，完成完整的数据库设计——E-R图、数据字典、DDL。

**我的方案**：
1. **ASCII画E-R图**：不用Mermaid（不是所有Markdown渲染器都支持），用ASCII box-drawing确保任何地方都能看
2. **8表数据字典**：每表列出字段/类型/约束/默认/说明五列，外加索引策略
3. **DDL脚本完整可执行**：含CREATE DATABASE、字符集utf8mb4、所有外键和索引、注释
4. **测试用例前置**：在设计阶段就列8条数据库测试，D7实现时直接转pytest

**为什么这样设计**：
- traffic_records使用BIGINT主键：评估日增数据近4000条（15min采样），INT在几个月后可能不够
- 外键全部CASCADE：路段删除时级联清理关联数据，避免孤儿记录
- coordinates用JSON：SUMO路网的坐标序列不定长，JSON比多行关联表更灵活
- prediction_results存置信区间而非单一值：让前端展示预测不确定性
- 预留partition策略（按月RANGE分区traffic_records），详细设计阶段实施

**风险**：
- DDL未经MySQL实际验证 → 开发阶段(D6)需要Leader在真实MySQL上测试
- 索引设计基于预估查询模式，实际性能需EXPLAIN验证
- coordinates JSON查询在MySQL 8.x中可用JSON_CONTAINS等函数，但可能影响索引效率
