## 一、完整业务逻辑链条

系统围绕"感知→预测→预警→决策→反馈"五环闭环运行：

```
┌─────────────────────────────────────────────────────────────────┐
│                    业务逻辑全链路                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [数据感知层]                                                    │
│  SUMO路网建模 → 车流需求定义 → E2检测器部署 → 微观仿真运行        │
│  → XML输出(流量/速度/占有率) → TraCI Python解析 → MySQL持久化     │
│                                                                 │
│  [数据加工层]                                                    │
│  缺失值填充 → 异常值检测(IQR/3σ) → Min-Max归一化                  │
│  → 时间特征提取(小时/星期/是否高峰/是否节假日)                     │
│  → 滞后特征构造(t-1, t-2, t-3) → 训练/测试集切分                  │
│                                                                 │
│  [预测引擎层]                                                    │
│  KNN回归(查找最相似历史时刻) + 随机森林(集成学习)                  │
│  → 双模型并行预测 → 拥堵等级分类(畅通/缓行/拥堵/严重拥堵)          │
│  → 模型评估对比(MAE/RMSE/MAPE) → Celery定时重训练                 │
│                                                                 │
│  [预警决策层]                                                    │
│  预测流量 > 阈值 → WebSocket实时推送 → 前端弹窗 + 声音提醒         │
│  → 管理员确认/处理/标记误报 → 预警记录入库 → 统计报表              │
│                                                                 │
│  [出行服务层]                                                    │
│  用户选点 → 加载实时路况权重 → Dijkstra最短时间路径               │
│  → 多路径对比(最短距离/最短时间/备用) → 高德地图渲染               │
│                                                                 │
│  [反馈闭环]                                                      │
│  预警处理结果 → 误报标记反馈 → 阈值调整建议                        │
│  模型评估指标 → 参数调优 → 模型迭代                               │
└─────────────────────────────────────────────────────────────────┘
```

## 三、角色定义与核心诉求

| 角色 | 核心职责 | 日常工作场景 | 关键诉求 |
|------|----------|-------------|----------|
| **交通管理员** | 全城态势感知 + 预警处置 + 调度决策 | 指挥中心大屏持续监控，收到预警后快速研判并下发指令 | "看得全、判得准、响应快" |
| **数据分析员** | 模型训练评估 + 数据深度分析 + 报告输出 | 定期检查模型精度，分析拥堵规律，为管理者提供数据支撑 | "模型准、分析深、可追溯" |
| **普通出行者** | 路线查询 + 出行规划 + 个性化提醒 | 出行前查路况、规划路线；日常通勤获取主动推送 | "路线优、提醒及时、操作简单" |

**关键交互原则:**

- 管理员和分析员走 **Web端**，大屏/宽屏布局，信息密度高
- 出行者走 **响应式页面**（适配手机浏览器），操作路径短，决策信息精简

## 六、核心业务流程（跨角色协作）

### 6.1 预警全生命周期流程

SUMO持续仿真 -> 每5min数据输出 -> 预测引擎(Celery定时) -> 预警引擎(Flask规则判断) -> 流量>阈值且速度<阈值 -> WebSocket推送给管理员 -> 管理员弹窗提示 -> 研判决策(确认处理/下发调度/标记误报) -> PUT处理结果 -> 更新预警状态写入操作日志 -> 若标记误报反馈给预测引擎调整阈值

### 6.2 分析员支撑管理员决策流程

管理员发现某区域经常拥堵 -> 分析员打开拥堵传播分析 -> 选取目标路段查看传播链 -> What-If仿真(限流/信号调优/分流) -> SUMO快速仿真返回对比报告 -> 输出分析结论+推荐方案 -> 管理员采纳执行 -> 更新信号配时/发布出行提示

### 6.3 出行者个性化服务流程

用户注册+首次路径规划 -> 连续3天查询同一OD对 -> 系统识别为高频路线写入画像表 -> 第4天7:30定时任务触发 -> 查询该路线预测数据 -> 预测拥堵 -> 推送消息 -> 用户收到推送 -> 点击跳转路径规划 -> 查看备选路线确认出发 -> 更新画像(记录用户接受建议)

## 九、核心新增功能说明

### 9.1 拥堵传播链分析 (P0)

- **输入**: 当前所有路段的拥堵等级 + 路网邻接矩阵（从 SUMO `.net.xml` 提取 edges 连接关系）
- **算法**: 基于图扩散模型，计算拥堵在 15/30/60 分钟后的传播范围和影响区域
- **前端**: 在地图上以涟漪动画展示拥堵扩散趋势，标注预计受影响路段
- **价值**: 让管理者从"单点处置"升级为"主动阻断传播链"

### 9.2 出行者个性化画像与主动推送 (P0)

- 用户登录后，系统记录其历史查询的 OD 对、常用出行时段
- 每日高峰前自动分析该用户常走路线的预测拥堵状态
- 若预测拥堵 → WebSocket 推送提醒，建议提前出行或切换路线
- 前端：新增"我的出行"页面，展示常用路线健康度卡片

### 9.3 信号灯配时优化建议 (P1)

- 基于路口各进口道的预测流量，使用 Webster 配时公式计算最优绿信比
- 展示当前固定配时 vs 优化建议配时的对比
- 公式: C_opt = (1.5L + 5) / (1 - Y)，L 为总损失时间，Y 为各相位最大流量比之和

### 9.4 应急车辆优先通行调度 (P1)

- 管理员在地图上标记应急车辆当前位置和目标位置
- 系统自动计算最优路径，并在沿线路口生成"绿波带"建议
- 应急路线以蓝色闪烁线条高亮，沿途路口标注建议放行方向

### 9.5 多场景 What-If 仿真对比 (P2)

- 场景 A（现状）：不做干预
- 场景 B（限流）：降低某入口 30% 车流
- 场景 C（分流）：引导车辆绕行
- 前端并排展示三种场景的关键指标对比（总延误、平均速度、拥堵路段数）

### 9.6 碳排放与环保看板 (P2)

- 基于路段平均速度 + 车辆类型分布估算 CO2 排放量
- 展示每日/每周排放曲线、拥堵导致的多余排放量
- 将"碳减排量"作为信号优化和拥堵疏导的效果度量
